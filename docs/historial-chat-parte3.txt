FASE 0: PLANIFICACION Y SETUP
------------------------------
- Exploracion del workspace D:\HelpDesk, encontrados grtc-helpdesk y pc-watchdog-api
- Decisiones del usuario: proyecto NUEVO desde cero, monolito Next.js, NextUI, solo gratis
- Stack: Next.js 14 + NextUI + Prisma/SQLite + Vitest + Playwright
- Proyecto creado con create-next-app, dependencias instaladas
- Prisma schema refinado (SQLite no soporta enums, autoincrement en no-id)
- Archivos fundacionales: DESIGN_TOKENS.md, domain-model.md, ADRs, spec Gherkin
- Estructura de carpetas: core/, lib/, modules/, docs/, __tests__/, e2e/

FASE 1: AUTH (LOGIN, REGISTER, LOGOUT)
---------------------------------------
- Middleware JWT con jose, cookie httpOnly, protege /dashboard/* y /api/*
- API routes: POST /api/auth/login, /api/auth/register, /api/auth/logout
- Zod schemas para validacion (LoginSchema, RegisterSchema)
- LoginForm con NextUI (email, password, link a register)
- RegisterForm con NextUI (nombre, email, password, rol Select)
- Seed con admin@taller.com y tecnico@taller.com
- Tests unitarios 12/12 pasando (auth actions + API routes)
- Tests E2E 22 escritos con Playwright (caja negra)
- Bugs corregidos: IconWrench→IconTool, colores dinamicos Tailwind, SelectItem value
- Login/register usan window.location.href (router.push no funcionaba en tests)

FASE 2: CRUD CUSTOMERS + DEVICES
---------------------------------
- API Customers: GET/POST /api/customers, GET/PUT/DELETE /api/customers/[id]
- API Devices: GET/POST /api/devices, GET/PUT/DELETE /api/devices/[id]
- Zod schemas: CustomerSchema, DeviceSchema con validaciones
- Customers page: tabla con busqueda, modal crear/editar (nombre, telefono, email)
- Devices page: tabla con busqueda, modal crear/editar (marca, modelo, serial, tipo, cliente selector)
- Validaciones: serial unico, cliente requerido, device pertenece al customer seleccionado

FASE 3: CORE TICKETING
-----------------------
- Tipos: CreateTicketSchema, UpdateTicketSchema, AssignTicketSchema
- VALID_STATUS_TRANSITIONS: RECEIVED→DIAGNOSING→REPAIRING/WAITING→READY→DELIVERED
- TicketNumber autoincremental (MAX+1)
- API Tickets: GET/POST /api/tickets, GET/PUT/PATCH /api/tickets/[id]
- API Users: GET /api/users (lista tecnicos para asignacion)
- Tickets page: tabla con busqueda + filtro por estado
  - Modal crear: cliente selector → device filtrado, descripcion, prioridad, costo
  - Modal detalle: info completa + historial + cambio de estado con botones
  - Modal asignar: selector de tecnico
- Historial en cada accion: CREATED, STATUS_CHANGE, ASSIGNMENT
- api-client actualizado para soportar PATCH
- Dashboard con datos reales de BD (4 cards)
- Technicians page con listado de usuarios

FASE 4: SEED DEMO + CIERRE DE TICKET
-------------------------------------
- Seed masivo con datos realistas:
  - 4 usuarios (admin + 3 tecnicos con especialidades)
  - 6 clientes (persona natural + empresa + colegio)
  - 10 equipos (LAPTOP, DESKTOP, ALL_IN_ONE, TABLET)
  - 12 tickets en TODOS los estados:
    2 RECEIVED, 2 DIAGNOSING, 2 REPAIRING, 2 WAITING_PARTS, 2 READY, 2 DELIVERED
- Cierre de ticket: READY → DELIVERED registra costo final en historial
- History: "Ticket entregado al cliente. Costo final: S/ XX.XX"
